# AIPPT PPT 制作方法

这份文档解释 AIPPT 当前如何从一句需求生成 PPTX，以及 Hermes/MiMo 在其中
负责什么、未来哪些经验会进入记忆。

## 总体原则

AIPPT 不是把一段 prompt 直接丢给模型，然后期待模型一次性吐出完整 PPT。我们把
流程拆成两层：

- Hermes/MiMo 负责理解需求、规划内容、组织讲述结构和沉淀偏好。
- 确定性 builder 负责解析大纲、选择版式、套用 SJTU 模板并生成可编辑 PPTX。

这样做的好处是：模型可以发挥规划能力，但最终 PPTX 的几何位置、模板页、文件
权限和产物记录仍由可测试的后端代码控制。

## 一次生成的完整流程

1. 用户在网页输入标题和需求，点击 `生成 PPT`。
2. API 创建 `DeckSession` 和 `plan_outline` 任务，并把用户输入写入独立 job
   workspace 的 `input/outline.md`。
3. Worker 领取任务，调用 Hermes/MiMo，并加载 `aippt-sjtu-ppt` skill。
4. Hermes/MiMo 把简短需求或已有大纲改写成 Markdown 规划稿，写入
   `ir/planned_outline.md`。
5. 后端把规划稿更新为 deck 的可编辑大纲，并自动排队 `build_pptx` 任务。
6. Builder 将 Markdown 大纲转换成 Deck IR，写入 `ir/deck.json`。
7. Builder 读取 SJTU 模板，生成封面、内容页、结束页，输出 `out/deck.pptx`。
8. API 记录 PPTX、Deck IR、日志等 `FileAsset`，网页出现 `下载 PPTX`。

用户看到的 1/2/3/4 进度条对应：

```text
理解需求 -> Hermes/MiMo 规划 -> PPTX 渲染 -> 完成下载
```

网页中的工作笔记来自 `AIPPT_AGENT:` 日志。它展示的是公开的执行摘要，例如
“正在规划页面：机器学习的边界与挑战”，不是模型的隐藏思维链。

## Hermes/MiMo 具体做什么

Hermes/MiMo 当前主要承担深度规划：

- 判断用户输入是一句话需求、半结构化大纲，还是完整页级大纲。
- 识别页数要求；例如“5-6 页”会被尊重，而不是自动扩成十几页。
- 压缩封面标题和副标题，避免把目录和说明文字塞进封面。
- 为每一页建立一个清楚的核心判断。
- 把正文组织成 2-3 个主题方向，每个方向有 2-4 个具体要点。
- 避免整页都是稀疏 bullet，优先让 builder 渲染成卡片、流程、事实块或公式块。
- 识别公式内容并保留为可编辑文本，例如 `J(θ)=1/m ∑ᵢ L(yᵢ, fθ(xᵢ))`。
- 避免固定标签污染页面，比如不再输出“一句话：”作为可见文字。
- 不生成最后的“谢谢”页，因为系统会统一套用模板结束页。

Hermes 输出仍然是 Markdown，而不是直接写 PPTX。这是刻意的边界：规划稿方便人类
检查，也方便 builder 做结构化解析和测试。

## Builder 具体做什么

Builder 是生产稳定性的核心。它把 Hermes 的 Markdown 规划稿变成 PPTX：

- `outline.py` 解析标题、页码、封面、正文、公式和 bullet。
- 解析结果会转换为受约束的 Deck IR，而不是自由坐标脚本。
- `render.py` 读取 SJTU 模板，使用模板封面、内容页 header 和结束页。
- 内容页按结构选择卡片、流程、三栏、公式提示框等版式。
- 正文和公式都以 PowerPoint 文本框形式存在，后续可编辑。
- 构建完成后写入 PPTX、IR、日志，并由 API 做用户隔离和下载鉴权。

这也是为什么 PPTX 渲染常常只需要一两秒：最慢的是 Hermes/MiMo 的规划，PPTX
文件生成本身是本地确定性代码完成的。

## 为什么不让 Hermes 直接生成 PPTX

短期不让 Hermes 直接接管 PPTX 文件生成，原因有四个：

- 稳定性：坐标、字体、模板占位符和文本溢出需要可测试代码控制。
- 安全性：Hermes 不应该决定用户权限、文件存储路径或下载鉴权。
- 可回滚：Markdown 和 Deck IR 都能审查、修复、重新构建。
- 可复用：同一份规划稿可以换模板、重新渲染或局部修复。

未来 Hermes 可以参与局部修复，比如根据 QA 报告修改某一页的 IR，但接受修复前仍
需要 builder validation 和回归测试。

## Hermes 会如何“记忆”和成长

当前系统已经加载 `memory` toolset 和 `aippt-sjtu-ppt` skill，但生产侧会谨慎处理
长期记忆。我们把记忆分成三类：

```text
template memory  产品级规则，例如 SJTU 红金配色、封面短标题、避免纯 bullet
group memory     课题组/课程偏好，例如组会分享要学术克制、结论先行
user memory      单个用户偏好，例如喜欢三栏卡片、正文密一点、少用营销语言
```

适合进入长期记忆的是“偏好和经验”，不是“私有内容全文”。

可以记：

- 用户反复要求“内容更详实，每张卡片 2-4 个 bullet”。
- 用户喜欢 SJTU 模板封面、内容页 header 和三卡片教学风格。
- 某个课题组偏好 DFT/MD/HPC 例子，语气要像组会分享。
- 用户明确说“以后都不要出现‘一句话’这个标签”。

不可以记：

- 私有 PPT 的完整大纲或上传文档全文。
- API key、cookie、路径、服务器地址、OAuth secret。
- 另一个用户的个人偏好。
- 含敏感信息的原始 job log。

现在最可靠的“成长”路径是：

1. 用户给出明确反馈。
2. Hermes 或后端把反馈转成小而干净的 memory candidate。
3. 标注作用域、置信度和证据来源。
4. 下一次规划时读取相应作用域的偏好。
5. 如果生成质量稳定提升，再把它固化到 skill 或 builder 规则。

## 当前已经沉淀下来的经验

这几轮开发已经变成了系统规则：

- 默认走 Hermes/MiMo 深度规划，而不是快速直接渲染。
- 规划完成后自动生成 PPTX，用户不需要第二次点击。
- 页面只显示可读工作笔记，不显示原始 server log。
- 封面标题和副标题必须短。
- 内容页优先卡片化，每个卡片应有标题和 2-4 个支撑点。
- 正文字号要适合投影阅读。
- 公式优先用可编辑文本。
- 三个卡片之间有编号时，不再额外画箭头。
- SJTU 模板的正文占位虚线框需要清理，不应出现在成品页。

这些规则一部分在 Hermes skill 中，一部分在 builder 代码和测试中。长期看，好的
经验应该进入三处：skill 负责规划习惯，memory 负责用户/课题组偏好，builder 负责
稳定可测试的视觉实现。
